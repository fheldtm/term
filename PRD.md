# SSH Terminal Composer PRD

## 1. 개요

이 프로젝트는 SSH 터미널을 중심으로, 좌측 파일 탐색기와 하단 채팅형 입력기를 결합한 데스크톱/웹 터미널 애플리케이션이다. 전반적인 사용 경험은 MobaXterm처럼 원격 서버에 접속한 뒤 파일 시스템을 탐색하고 터미널을 사용하는 구조를 따른다.

핵심 차별점은 일반 터미널 입력 대신, 이미지와 파일을 본문 안의 원하는 위치에 삽입할 수 있는 채팅형 composer를 제공하는 것이다. 사용자는 `Ctrl+V`, 드래그앤드롭, 파일 선택 버튼으로 이미지나 파일을 추가하고, composer 상단의 Preview 영역에서 첨부를 확인한 뒤 제출할 수 있다.

제출 시 앱은 첨부 파일을 서버의 지정된 업로드 경로로 전송하고, 본문 안의 첨부 토큰과 파일 목록을 포함한 텍스트 payload를 SSH 터미널 입력으로 보낸다.

## 2. 목표

- 사용자가 SSH 세션에 접속한 상태에서 터미널, 파일 탐색기, 첨부 가능한 입력기를 한 화면에서 사용할 수 있게 한다.
- 사용자가 이미지/파일을 입력 본문 중 원하는 위치에 삽입할 수 있게 한다.
- 첨부된 이미지/파일을 서버에서 접근 가능한 경로에 저장하고, 해당 경로를 터미널에 전달되는 메시지에 포함한다.
- 이미지 첨부는 composer 상단 Preview 영역에서 미리 볼 수 있게 한다.
- 최종 제출 결과는 사람이 읽기 쉬운 텍스트 형태로 터미널에 입력되게 한다.

## 3. 비목표

- 자체 AI 모델 실행 또는 AI 응답 UI 구현은 MVP 범위가 아니다.
- 원격 서버의 모든 파일 편집 기능을 VS Code 수준으로 제공하지 않는다.
- 여러 SSH 세션 동시 관리, 세션 매크로, 포트 포워딩, X11 포워딩은 MVP 범위가 아니다.
- 파일 동기화, diff, git UI는 MVP 범위가 아니다.

## 4. 주요 사용자

- SSH로 원격 서버에 접속해 개발, 운영, 디버깅 작업을 하는 사용자
- 터미널 기반 AI 도구, CLI 에이전트, 스크립트에 이미지/파일 컨텍스트를 함께 전달하고 싶은 사용자
- 로컬에서 캡처한 이미지나 파일을 원격 서버 작업 문맥에 빠르게 포함해야 하는 사용자

## 5. 핵심 UX 구조

화면은 크게 세 영역으로 구성한다.

1. 좌측 파일 탐색기
   - 원격 서버의 현재 디렉터리 트리를 표시한다.
   - 폴더 확장/접기, 경로 이동, 새로고침을 지원한다.
   - 터미널의 현재 작업 디렉터리와 동기화할 수 있다.

2. 우측 터미널 영역
   - `xterm.js` 기반 터미널을 표시한다.
   - SSH PTY 입출력을 연결한다.
   - 사용자가 직접 터미널에 입력할 수도 있고, composer 제출 payload도 동일한 SSH 입력 스트림으로 전달된다.

3. 우측 하단 composer 영역
   - 텍스트 입력, 첨부 추가, 이미지 Preview, 제출 버튼을 포함한다.
   - 첨부된 파일은 본문 안에서 `[_image1]`, `[_file1]` 같은 토큰으로 참조된다.
   - 제출 시 본문과 첨부 파일 목록을 합쳐 터미널에 전송한다.

## 6. Composer 요구사항

### 6.1 텍스트 입력

- 사용자는 일반 채팅창처럼 여러 줄 텍스트를 입력할 수 있어야 한다.
- 입력창은 최소 높이를 가지며, 내용이 길어지면 일정 높이까지 확장한 뒤 스크롤된다.
- `Enter`는 기본적으로 줄바꿈으로 동작한다.
- 제출 단축키는 `Ctrl+Enter` 또는 `Cmd+Enter`를 기본값으로 한다.

### 6.2 첨부 추가 방식

사용자는 다음 방식으로 파일을 추가할 수 있어야 한다.

- 클립보드 붙여넣기
  - `Ctrl+V` 또는 `Cmd+V`로 이미지 파일을 붙여넣는다.
  - 클립보드에 이미지 바이너리가 있으면 이미지 파일로 변환해 첨부한다.

- 드래그앤드롭
  - 파일 또는 이미지를 composer 영역에 드롭한다.
  - 드롭한 위치를 계산할 수 있으면 해당 위치에 토큰을 삽입한다.
  - 위치 계산이 어렵다면 현재 커서 위치에 삽입한다.

- 파일 선택 버튼
  - composer 툴바의 첨부 버튼으로 로컬 파일 선택창을 연다.
  - 선택된 파일은 현재 커서 위치에 토큰으로 삽입된다.

### 6.3 첨부 토큰 규칙

- 이미지는 `[_imageN]` 형식으로 삽입한다.
- 일반 파일은 `[_fileN]` 형식으로 삽입한다.
- `N`은 composer 내에서 1부터 증가하는 번호다.
- 토큰은 사용자가 본문 내에서 복사, 이동, 삭제할 수 있다.
- 동일 첨부 파일을 여러 위치에서 참조하는 것은 허용한다.
- 제출 시 본문에 존재하지 않는 첨부 토큰은 경고 상태로 표시한다.
- Preview에는 있지만 본문에서 참조되지 않는 첨부는 제출 전 사용자에게 표시한다.

예시:

```text
[_image1] 이 화면을 보고 수정해줘. 특히 [_image2] 부분이 이상해.
아래 로그도 같이 참고해줘. [_file1]
```

### 6.4 Preview 영역

- composer 입력창 위쪽에 첨부 Preview 영역을 둔다.
- 이미지 파일은 썸네일, 파일명, 삭제 버튼을 표시한다.
- 일반 파일은 파일 아이콘, 파일명, 크기, MIME type, 삭제 버튼을 표시한다.
- 이미지 썸네일 클릭 시 큰 미리보기 또는 별도 팝오버를 제공할 수 있다.
- Preview의 각 항목에는 연결된 토큰을 표시한다.

예시:

```text
[_image1] image.png
[_image2] screenshot.png
[_file1] server.log
```

### 6.5 첨부 삭제

- Preview에서 첨부 삭제 시 해당 첨부의 토큰을 본문에서도 제거한다.
- 본문에서 토큰을 직접 삭제하면 Preview 항목은 남기되 "본문에서 참조되지 않음" 상태로 표시한다.
- 사용자는 참조되지 않은 Preview 항목을 다시 본문에 삽입하거나 삭제할 수 있다.

### 6.6 제출 버튼

- 제출 버튼을 누르면 다음 순서로 처리한다.
  1. 본문과 첨부 상태를 검증한다.
  2. 첨부 파일을 원격 서버의 업로드 경로로 전송한다.
  3. 원격 파일 경로가 포함된 최종 payload를 생성한다.
  4. payload를 SSH PTY 입력으로 보낸다.
  5. composer 입력과 첨부 상태를 초기화한다.

- 업로드 실패 시 터미널에 payload를 보내지 않는다.
- 실패한 파일과 실패 사유를 composer에 표시한다.

## 7. 제출 Payload 포맷

터미널에 전달되는 최종 payload는 다음 형식을 따른다.

```text
[_image1] 채팅 내용 ... [_image2] ... 채팅 [_image3] [_image4]

User uploaded files:
- [_image1] image.png (593.7 KB, image/png): /remote/upload/path/image.png
- [_image2] image.png (137.9 KB, image/png): /remote/upload/path/image-2.png
- [_image3] image.png (98 KB, image/png): /remote/upload/path/image-3.png
- [_image4] diagram.png (220 KB, image/png): /remote/upload/path/diagram.png
```

규칙:

- 본문을 먼저 출력한다.
- 첨부가 하나 이상 있을 때만 `User uploaded files:` 섹션을 추가한다.
- 목록은 토큰 번호 순서대로 정렬한다.
- 파일 크기는 사람이 읽기 쉬운 단위로 표시한다.
- 경로는 원격 서버 기준의 절대 경로를 사용한다.
- 마지막에는 터미널에서 제출되도록 newline을 포함한다.

## 8. 원격 업로드 요구사항

### 8.1 업로드 경로

- 기본 업로드 경로는 원격 서버의 홈 디렉터리 하위 폴더로 한다.

```text
~/.terminal-composer/uploads/YYYY-MM-DD/session-id/
```

- 실제 payload에는 쉘 축약 경로가 아닌 절대 경로를 넣는다.

예시:

```text
/home/fheldtm/.terminal-composer/uploads/2026-05-25/abc123/image.png
```

### 8.2 파일명 정책

- 원본 파일명을 최대한 유지한다.
- 동일 이름 충돌 시 suffix를 붙인다.

```text
image.png
image-2.png
image-3.png
```

- 파일명에는 경로 구분자, 제어 문자, 위험 문자를 허용하지 않는다.

### 8.3 업로드 방식

- SSH 연결과 함께 SFTP 채널을 사용한다.
- SFTP가 불가능한 환경에서는 SCP 또는 base64 stream 업로드를 fallback으로 고려한다.
- MVP에서는 SFTP 업로드를 기본 요구사항으로 한다.

## 9. 파일 탐색기 요구사항

- SSH 접속 후 원격 홈 디렉터리를 기본 루트로 표시한다.
- 폴더 확장 시 해당 경로의 하위 파일/폴더를 lazy load한다.
- 파일/폴더명, 타입, 크기, 수정일을 내부 데이터로 관리한다.
- 사용자에게는 MobaXterm처럼 조밀하고 탐색 효율이 높은 트리 UI를 제공한다.
- 파일 더블클릭 동작은 MVP에서 다음 중 하나로 제한한다.
  - 텍스트 파일 미리보기
  - 경로를 composer에 삽입
  - 터미널에 `cd` 또는 `cat` 명령 삽입

MVP에서는 "경로를 composer에 삽입"을 우선한다.

## 10. 터미널 요구사항

- `xterm.js`를 사용한다.
- SSH PTY와 연결해 실시간 입출력을 처리한다.
- 터미널 크기 변경 시 PTY resize 이벤트를 보낸다.
- composer 제출 payload는 화면 출력용 `xterm.write()`가 아니라 SSH 입력 스트림으로 전달한다.
- 사용자가 일반 터미널처럼 직접 입력할 수 있어야 한다.

## 12. 주요 사용자 플로우

### 12.1 이미지 붙여넣기 후 제출

1. 사용자가 composer에 텍스트를 입력한다.
2. 사용자가 `Ctrl+V`로 이미지를 붙여넣는다.
3. 현재 커서 위치에 `[_image1]` 토큰이 삽입된다.
4. Preview 영역에 이미지 썸네일이 표시된다.
5. 사용자가 제출 버튼을 누른다.
6. 이미지가 원격 업로드 경로로 전송된다.
7. 본문과 `User uploaded files:` 목록이 터미널에 입력된다.

### 12.2 여러 이미지를 원하는 위치에 삽입

1. 사용자가 본문 중간에 커서를 둔다.
2. 이미지 A를 드롭한다.
3. 해당 위치에 `[_image1]`이 들어간다.
4. 사용자가 다른 문장 뒤에 커서를 둔다.
5. 이미지 B를 붙여넣는다.
6. 해당 위치에 `[_image2]`가 들어간다.
7. 제출 payload는 본문 위치를 그대로 보존한다.

### 12.3 Preview에서 첨부 삭제

1. 사용자가 Preview의 `[_image2]` 삭제 버튼을 누른다.
2. 첨부 목록에서 이미지가 제거된다.
3. 본문 안의 `[_image2]` 토큰도 제거된다.
4. 남은 첨부 토큰 번호는 재정렬하지 않는다.

## 13. 에러 처리

- SSH 연결이 끊긴 상태에서 제출하면 제출을 막고 재연결 안내를 표시한다.
- 업로드 경로 생성 실패 시 제출을 막는다.
- 파일 업로드 실패 시 실패한 첨부를 Preview에 표시한다.
- 지원하지 않는 클립보드 데이터는 무시하지 말고 간단한 메시지로 알려준다.
- 파일 크기가 제한을 초과하면 첨부 전에 차단한다.
- 본문에 존재하는 토큰이 첨부 목록에 없으면 제출 전 경고한다.

## 14. 제한 정책

초기 기본값:

- 첨부 파일 최대 개수: 20개
- 단일 파일 최대 크기: 50 MB
- 전체 첨부 최대 크기: 200 MB
- 이미지 Preview 대상 MIME type: `image/png`, `image/jpeg`, `image/webp`, `image/gif`

이 값은 설정에서 변경 가능하도록 설계한다.

## 15. 접근성 및 키보드 UX

- composer는 키보드만으로 파일 첨부, Preview 탐색, 삭제, 제출이 가능해야 한다.
- Preview 삭제 버튼은 명확한 aria-label을 가진다.
- 드롭 영역은 파일 드래그 중 시각적으로 강조된다.
- 제출 중에는 버튼 로딩 상태를 표시하고 중복 제출을 막는다.

## 16. 보안 고려사항

- 로컬 파일 경로를 payload에 노출하지 않는다.
- payload에는 원격 서버 경로만 포함한다.
- 파일명은 sanitize한다.
- 원격 업로드 경로는 현재 SSH 사용자 홈 디렉터리 하위로 제한한다.
- 터미널에 보내기 전 payload preview를 내부적으로 만들되, 사용자가 의도하지 않은 shell command escaping을 자동으로 변경하지 않는다.
- 사용자가 입력한 텍스트는 원문을 보존한다.

## 17. MVP 범위

MVP에 포함:

- 단일 SSH 세션 연결
- `xterm.js` 터미널
- 원격 파일 탐색기 기본 트리
- composer 텍스트 입력
- 이미지/파일 붙여넣기, 드래그앤드롭, 파일 선택
- 본문 내 첨부 토큰 삽입
- Preview 영역
- SFTP 업로드
- 최종 payload 생성 및 SSH PTY 입력 전송

MVP 이후:

- 다중 세션/탭
- rich text 토큰 editor
- 파일 탐색기에서 업로드/다운로드
- 첨부 재사용 라이브러리
- payload 템플릿 커스터마이징
- 원격 파일 미리보기
- 세션별 업로드 폴더 관리 UI

## 18. 수용 기준

- SSH 접속 후 터미널에서 일반 명령을 입력하고 결과를 볼 수 있다.
- 좌측 파일 탐색기에서 원격 홈 디렉터리의 파일/폴더를 확인할 수 있다.
- composer에 텍스트를 입력할 수 있다.
- 이미지를 붙여넣으면 현재 커서 위치에 `[_image1]` 토큰이 삽입된다.
- 붙여넣은 이미지는 composer 상단 Preview에 썸네일로 표시된다.
- 여러 이미지를 추가하면 `[_image1]`, `[_image2]` 순서로 토큰이 생성된다.
- 사용자는 본문 안에서 토큰의 위치를 이동할 수 있다.
- 제출 시 첨부 파일이 원격 서버 업로드 경로에 저장된다.
- 제출 시 터미널에는 본문과 `User uploaded files:` 목록이 함께 입력된다.
- 업로드 실패 시 터미널로 payload가 전송되지 않고 에러가 표시된다.

## 19. 결정 및 미결정 사항

- 앱 형태: 브라우저 기반 웹앱으로 구성한다.
- SSH 구현 위치: Node.js 백엔드가 SSH/SFTP 프록시 역할을 맡는다.
- 파일 탐색기의 루트 범위: 원격 홈 디렉터리만 보여줄지 전체 파일 시스템 접근을 허용할지 결정해야 한다.
- Preview 이미지 클릭 시 동작: 확대 보기, 원본 열기, 토큰 위치 이동 중 선택해야 한다.
- 제출 전 payload 확인 모달을 제공할지 결정해야 한다.
